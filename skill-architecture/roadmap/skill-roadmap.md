# Skill 开发路线图 (Roadmap) v1.0

**版本**: v1.0
**周期**: 2026 Q3 — 2027 Q4 (6 季度)
**北极星指标**: 真实用户反馈 + Skill 复用率
**关键产出**: L2 模式蓝图 4 个 / L3 场景 30+ / L4 多Agent 10+ / 行业蓝图 7+

---

## 1. 路线图核心思想

> **架构先行，复用驱动，自下而上验证**

不再"看到一个需求做一个 Skill"，而是先把**底座（L0/L1/L2）**夯实，让上层场景（L3/L4）通过组合现有构件得到，**避免重复造轮子**。

### 1.1 三个北极星
1. **复用率**：L3/L4 场景中 70%+ 步骤来自已有 L0-L2
2. **质量门禁通过率**：新增 Skill 一次通过 G1-G7 比例 > 80%
3. **真实落地**：每个季度至少 1 个真实客户场景上线

### 1.2 四个不做
- ❌ 不做"假大空"Skill（没有真实业务驱动）
- ❌ 不为 KPI 数量做 Skill（宁可少做精做）
- ❌ 不写空壳（必须可演示，必须有验证步骤）
- ❌ 不做无法治理的 Skill（绕过质量门禁）

## 4. 优先级矩阵

二维评估每个候选 Skill：

```
价值高 │  P0 必做       │  P1 应做       │
       │  (做了就赚)     │  (做了更好)    │
       ├─────────────────┼────────────────┤
价值低 │  P2 可做       │  ❌ 不做        │
       │  (有空再做)     │  (时间陷阱)    │
       └─────────────────┴────────────────┘
        实施成本低         实施成本高
```

判定准则：
- 价值高：客户主动提需求 / 解决真实痛点 / 有可量化业务价值
- 价值低：技术驱动 / "好像有用" / 短期内无客户
- 成本低：复用 ≥ 60% 现有 L0-L2 / 1 周内可上线
- 成本高：需要全新连接器 / 需要新算法 / 跨系统集成

## 5. 季度评审机制

### 5.1 季度初
- 回顾上季度 OKR 完成度
- 收集客户反馈（飞书多维表格 + 走访）
- 用 inventory-scan 出基线报告
- 制定本季度 OKR

### 5.2 季度末
- 跑 inventory-scan + skill-lint 出质量报告
- 评估 P0 完成率（< 80% 触发反思）
- 复盘：哪些 Skill 被复用 ≥ 5 次（赢家），哪些没人用（败者）
- 删除 / 归档 3 个月内零调用且无客户依赖的 Skill

### 5.3 反熵机制
每季度清理：
- 重复 Skill（合并到一个）
- 无 frontmatter 的"野生" Skill（强制补或删）
- 失效 Skill（依赖的外部API已下线）
- 文档与实现不一致的 Skill（修齐或删）

## 7. 风险与对冲

| 风险 | 触发信号 | 对冲措施 |
|------|----------|----------|
| Skill 数量虚高 | 季度新增 50+ 但复用率 < 30% | 暂停新增 1 月，专注治理 |
| 客户场景对不上架构 | 客户需求 6 周内无法对应到现有 L2 | 召开架构 RFC，调整分层 |
| 节点能力失衡 | 某节点持续 2 季度无新交付 | 重新分配责任 / 转角色 |
| 质量门禁形同虚设 | 通过率 100% 但客户反馈差 | 增强 G5/G6/G7 检查 + 引入用户验收门禁 |
| 平台能力受限 | TeleAgent / 小龙虾 平台不支持某能力 | 在 L0 适配器层做 polyfill 或降级 |

## 10. 一旦偏离路线图...

**触发条件**：连续 2 季度未达 OKR 70% 或客户反馈持续负面。

**修正流程**：
1. 暂停 P1/P2，专注 P0
2. 重新走访 3-5 客户，校准方向
3. 重写下季度 OKR，简化目标
4. 必要时重整节点责任分工

> 路线图是地图，不是圣经。客户真实反馈优先级最高。
